# Какую проблему решает
Proxy решает проблему контроля доступа к объекту без изменения самого объекта
он нужен когда
- прямой доступ к объекту 
	- дорогой
	- небезопасный
	- нежелательный
- требуется
	- ленивое создание
	- защита
	- кэш
	- удаленный доступ
- клиент не должен отличать реальный объект от прокси

---
# Симтомы
- необходимость 
	- проверять права перед каждым вызовом
	- лениво инициировать объект
	- контролировать доступ к ресурсу
- дорогие операции
	- сетевые вызовы
	- создание тяжелых объектов


---
# Суть
- proxy  реализует тот же интерфейс что и реальный объект
- Содержит ссылку на реальный Subject
- решает
	- вызывать ли реальный объект
	- как именно его вызывать
- может
	- изменить момент вызова
	- изменить окружение вызова


---
# Ключевая идея
Подменять реальный объект его представителем, не меняя клиентский код
Клиент
- работает с интерфейсом
- не знает, реальный это объект или прокси

---
# Когда оправдан
когда нужен
- virtual proxy - lazy laoding
- protection proxy - авторизация
- Remote proxy - Удаленный сервис
- logging / monitoring proxy

---
# Когда антипаттерн
Если
- используется для добавления бизнес логики
- становится декоратором "все подряд"
- поведение объекта меняется неожиданно
- клиенту все же приходится знать что это прокси
- добавляет сложность без реальной пользы
