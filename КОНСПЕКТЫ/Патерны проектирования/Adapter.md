# Какую проблему решает
Adapter решает проблему несовместимости интерфейсов

он нужен когда:
- есть готовый код
- его интерфейс не совпадает с тем, оторый ожидает твоя система
- ты не можешь или не хочешь менять этот код

---
# Симтомы
- постоянные преобразования данных(mappers)
- бизнес код знает про:
	- форматы внешних API
	- странные имена полей
- зависимости от
	- конкретных SDK
	- legacy интерфейсов
- if/switch по типу интеграции
- тяжело заменить одну реализацию другой


---
# Суть
- он оборачивает существующий объект
- реализует ождаемый интерфейс
- внутри делегирует вызовы адаптируемому объекту
- при необходимости 
	- преобразует данные
	- скрывает детали протокола

---
# Ключевая идея
Отделить "что мне нужно" от "как это реализованно"
Adapter
- защищает домен от
	- внешних форматов
	- чужих конктрактов
- локализует изменения
	- поменялся API -> меняется только adapter


---
# Когда оправдан
- интеграция с 
	- внешними API
	- SDK
	- legacy кодом
- реализуешь
	- port & adapters
- доменная модель не должна знать про внешний мир


---
# Когда антипаттерн
- когда содержит бизнес логику
- адаптирует внутренние интерфесы друг к другу
- используется для маскировки плохого дизайна
- цепочка адаптеров становится слишком глубокой
"Непонятно где кончается адаптер и начинается бизнес"
