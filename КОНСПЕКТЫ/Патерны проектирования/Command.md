# Проблема которую решает
Command решает проблему жесткой связности между инициатором действий и его реализацией.
Без command:
- код который знает что делать, знает как именно это делать
- изменения в бизнес логике тянут изменения в контроллерах, сервисах, воркерах
- сложно
	- логировать действия
	- откладывать выполнения
	- повторять ретраить
	- выстраивать цепочки операций
Command выносит действие как бизнес операцию в отдельную сущность

# Симптомы
- метод сервисов вызывается из разных мест( http, cron, sage, queue)
- бизнес действие 
	- должно логироваться 
	- должно повторяться
	- может выполняться асинхронно
- контроллеры знают слишком много о бизнес логике
- есть понятие операции, а не просто метода
# Суть
Запрос на выполнения действия представляется объектом
# Ключевая идея
отделить:
- кто инициирует дейстие
	от
- кто и как его выполняет
И рассматривать действие как первоклассную сущность домена
Command это не просто вызов метода, это намерение изменить систему.
# Когда оправдан
- CQRS
- есть
	- очереди
	- фоновые задачи
	- distributes systems
- требуется
	- retry
	- idemptotency
	- audit trail
- действие бизнес операция а не crud
# Когда антипаттерн
- команда просто прокси
- нет
	- логирования
	- асинхронности
	- истории
- каждая команда = 1 метод сервиса
- Command используется ради красоты архитектуры
- система маленькая и не эволюционирует
