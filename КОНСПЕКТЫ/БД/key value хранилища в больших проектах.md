# Ключ значение: принципы
- значение произвольной структуры, доступ по уникальному ключу
- базовые операции get/set/delete за O(1) или логарифмическая
- нет сложных запросов, слабая или отсутствующая поддержка транзакций
- инкременты и "блокировки"
- а также вспомагательные структуры и операции поверх k/v
- кластер "из коробки" с (относителньо) горизонтальным масштабированием
- часто доступность в ущерб согласованости, но бывает наоборот (FoundationDB)
# Базовые исторические сценарии k/v - кэш
- Мультиплексирует соединения
- Легкий протокол: нет SQL
- Легкий внутри: простые структуры данных, нет транзакций, мало блокировок, все гарантированно в памяти = очень высокий лэтенси
- как результат: умеет C100k и ~100K rps на ядро
	- может миллион запросов в секунду для тысячи одновременных соединенний
# Типичные сценарии использования
- Кэш - memcached, Redis/Valkey, Dragonfly
- Персистентное хранилище с понижеными гарантиями - Redis
- Персистентное хранилище с повышенными гарантиями - Redis Cluster, Tarantool/Picodata, Cassandra/ScyllaDB, Aerospike, FoundationDB
- Распределенное хранилище конфигурации сервисов с высокими гаранитями: etcd/consul
- Хранилище объектов/документов - MongoDB, Redis
- Специализированные сервисы / data grids: Tarantool/Picodata, Redis
- SQL поверх k/v - Picodata, движки RocksDB/LevelDB
# Основные плюсы k/v
- Горизонтальная масштабируемость и отказоустойчивость
- Простота и гибкость
- Легкость встраивания и использования
- Производительность простые структуры, меньше блокировок, мультиплексирование

# Основные минусы
- Ограниченые типы запросов: point select или специализированные гоотовые структуры данных: нет джойнов, привычной аналитики
- Ограниченный скейлинг по ядрам
- Ограниченные гарантии надежности, изоляции и консистентнотси
